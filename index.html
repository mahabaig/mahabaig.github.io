AZURE SOC LAB WALKTHROUGH

This project demonstrates hands-on experience in building a Microsoft Azure SOC environment, analyzing security events, and visualizing global attack data.

Step 1: Environment Setup

I created a free Microsoft Azure subscription and deployed a Windows 10 VM honeypot exposed to the internet via RDP (3389). To attract real attack traffic, I disabled certain protections like the firewall and Network Level Authentication, allowing brute-force attempts to be logged and observed.

Step 2: Data Collection & Monitoring

The VM was connected to a Log Analytics Workspace (LAW) and integrated with Microsoft Sentinel. From there, I collected and analyzed SecurityEvent logs, particularly Event ID 4625 (failed logon attempts). This data stream allowed me to simulate how SOC analysts triage incoming telemetry in real-time environments.

Step 4: Results & Insights

Within just 24 hours, my VM recorded over 30,000 failed login attempts from multiple regions — including Poland, Netherlands, Belgium, and Argentina. The data revealed common brute-force tactics such as repeated username cycling and predictable password patterns. Visualizing these results through Sentinel helped me see how attacks scale globally and how a SOC prioritizes threats by frequency and origin.

Step 5: Key Takeaways

  • Configured and managed a full SIEM pipeline within Azure.
  • Strengthened KQL querying skills for event correlation and enrichment.
  • Gained experience in threat intelligence visualization using Sentinel maps and workbooks.
  • Developed a deeper understanding of real-world brute-force behavior and log interpretation.

Through this project, I not only replicated a professional SOC workflow but also learned how to interpret raw telemetry and transform it into actionable intelligence — skills directly applicable to security operations analysis and incident response.

ACCESS CONTROL & SECURITY AUDITING LAB WALKTHROUGH

This lab demonstrates how password-guessing and post-exploitation techniques reveal weak authentication controls — and how stronger policies such as MFA can mitigate such attacks. All testing was performed in the Virginia Tech Cyber Range Cyber Basics environment.

Step 1: Hydra Brute-Force Attack

Using Hydra on a Kali Linux VM, I executed an online password-guessing attack targeting the ftp.example.com service. Hydra was configured with four threads and the John the Ripper password list to identify weak credentials for user alice.

hydra -t 4 -l alice -P /usr/share/john/password.lst -vV ftp.example.com ftp

The tool made over 3,500 attempts and successfully cracked simple passwords — demonstrating the risk of short, predictable credentials.

Step 2: FTP Login Validation

After cracking the password, I authenticated to the FTP service and validated remote access to the UNIX account.

Step 3: Directory Enumeration and File Retrieval

With valid credentials, I enumerated directories and retrieved a file (flag.txt) to simulate data exfiltration.

Step 4: Expanded Attack Surface Testing

I repeated the technique against another user and observed similar weak-password behavior (reused patterns), underscoring the risk of poor password hygiene.

Step 5: Defensive Controls & Recommendations

  • Enforce multi-factor authentication (MFA) for all external services.
  • Require complex password policies (e.g., minimum 12 characters, no dictionary words).
  • Implement account lockout policies to prevent repeated login attempts.
  • Monitor authentication logs for abnormal volume or geographic anomalies.

Through this lab, I learned how attackers systematically exploit weak credentials — and how layered authentication defenses drastically reduce exposure. These techniques strengthen both blue-team defensive analysis and red-team testing skills.

NETWORK TRAFFIC ANALYSIS & INTRUSION DETECTION LAB WALKTHROUGH

This lab demonstrates how to capture, filter, and analyze packets using TCPDump and Wireshark to detect anomalies such as ARP scans and RDP traffic patterns.

Step 1 – Initializing TCPDump

Executed tcpdump -i 1 -c 10 on Kali Linux to capture live packets, revealing ARP requests and TCP handshakes between hosts.

Step 2 – Filtering by Protocol & Port

Applied filters (arp, dst port 3389) to isolate RDP traffic and confirm remote connections within the lab network.

Step 3 – Writing and Reading Capture Files

Saved captures to a .pcap file (-w) and re-read them (-r) for analysis — simulating forensic evidence collection.

Step 4 – Capturing Live Traffic in Wireshark

Launched Wireshark and captured HTTP traffic while browsing. Used filters like http.host and ip.addr == 10.1.66.165 to observe outgoing requests.

Step 5 – Following Streams

Used Follow → HTTP Stream to reconstruct client-server conversations and inspect headers for GET/POST methods and responses.

Step 6 – Detecting ARP Scans & Suspicious Activity

Filtered arp packets and ran nmap -sn 10.1.66.165/20 to observe broadcast requests. Detected “Who has … Tell …” messages indicating an ARP scan.

Key Insights & Takeaways

  • Captured and analyzed 50+ packets across multiple protocols (ARP, TCP, HTTP).
  • Detected ARP scan patterns and RDP traffic anomalies in real time.
  • Practiced evidence collection and packet-level inspection with Wireshark & TCPDump.
  • Enhanced skills in network forensics and intrusion detection for SOC and IR roles.
